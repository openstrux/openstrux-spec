# v0.6.0 Change Proposals ŌĆö Consolidated Review

Source: `openstrux` worktree `chore/v0.6.0-planning`, branch `openspec/changes/v0-6-0-*`.

**9 change packages** organized in four tracks:

| Track | Changes | Purpose |
|---|---|---|
| Core compiler | 1-5 | parse -> resolve -> validate -> lock -> manifest |
| Target generation | 6-7 | AST -> runnable TypeScript |
| Evidence | 8 | benchmarks + scorecard |
| Acceptance/demo | 9 | starter repo + end-to-end proof |

**Dependency spine:**
```
1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7 -> 8 -> 9
```

**Key decisions from architecture review:**
- Split validator-manifest into validator-config (semantic layer) + manifest-explain (compiled-output layer)
- Added lock-determinism between them (ADR-014)
- Added policy resolution plumbing to validator-config (ADR-010)
- Added normative generator spec to target-ts
- Clarified Tier 1/2 exit criteria in rods-target-ts
- Benchmarks: 10 simple use-cases + generation comparison (tokens + time) ŌĆö **alpha-only evidence** (full gate requires 20+ cases, 3+ LLM families)
- Grant-workflow: starter repo with archived changes, two prompt sets, scripts, results viewer
- Toolchain-bootstrap: AST already exists ŌĆö scope is review/evolve, not implement from scratch
- `@type` keyword throughout (per spec grammar), not `@strux`
- ADR-016 (Token-Efficient Authoring) created as Open, pending v0.6.0 measurements
- Two distinct hash identities: `sourceHash` in snap.lock (input identity) and `artifactHash` in mf.strux.json (output identity), linked by `lockRef`

## 1. Toolchain Bootstrap (`v0-6-0-toolchain-bootstrap`)

**Track:** Core compiler | **Repo:** openstrux-core + openstrux-spec

### Why

Openstrux-core has partial implementation ŌĆö `packages/ast/` contains v0.5.0-alpha.0 typed AST/IR definitions (7 source files, ~1,150 lines), but the monorepo has no build system, no test runner, and no conformance fixtures. The remaining five packages (parser, validator, manifest, lock, conformance) are empty stubs. Before any parsing, validation, or target generation can happen, the monorepo needs a working build system and the existing AST must be reviewed against the current v0.5.0 spec and evolved where needed.

### What changes

- pnpm workspace configuration, root `tsconfig.json`, vitest, turbo build runner
- **Review and evolve** existing `packages/ast/` against `openstrux-spec@0.5.0`:
  - Verify all node types in `common.ts`, `types.ts`, `values.ts`, `expressions.ts`, `access.ts`, `panel.ts` match current spec
  - Add any missing node types or fields
  - Ensure `@type` keyword alignment (grammar uses `@type` for record/enum/union)
  - Verify `SourceFile` top-level container
- Five valid `.strux` conformance fixtures for P0 domain model

### Capabilities

| Capability | Type | Description |
|---|---|---|
| `monorepo-toolchain` | New | pnpm workspace, tsconfig, vitest, turbo |
| `conformance-fixtures-p0` | New | 5 valid `.strux` fixtures for P0 |
| `core-ast` | Modified | Review/evolve v0.5.0-alpha.0 AST against current spec |

### Impact

- **openstrux-core**: Monorepo config + AST review/update
- **openstrux-spec**: 5 new valid fixtures
- Unblocks: parser, validator, generator

## 3. Validator + Config (`v0-6-0-validator-config`)

**Track:** Core compiler | **Repo:** openstrux-core + openstrux-spec

### Why

Parser makes no semantic checks. The validator enforces the type system and certification rules (ADR-011). Config inheritance must be resolved before validation. Policy resolution (ADR-010) is structural ŌĆö AccessContext and policy provenance affect whether the validator accepts a panel.

This is the semantic prerequisite layer. Manifest and explanation sit on top (see change 5).

### What changes

- `packages/validator/` ŌĆö type resolution, snap chain checking, AccessContext enforcement, scope validation, `@cert` enforcement (ADR-011), policy resolution diagnostics
- `packages/config/` ŌĆö `strux.context` cascade: `@dp` merge, `@access` narrowing, `@ops`/`@sec` merge, named `@source`/`@target` resolution, `@cert`-in-context rejection
- Policy resolution: guard evaluation tiers (inline -> hub -> external), `W_POLICY_OPAQUE` when policy source is unverifiable, `W_SCOPE_UNVERIFIED` when scope fields cannot be statically confirmed
- Conformance fixtures for config inheritance and ADR-011

### Capabilities

| Capability | Type | Description |
|---|---|---|
| `validator` | New | Type checking, knot compatibility, AccessContext, `@cert` (ADR-011), policy diagnostics |
| `config-inheritance` | New | `strux.context` cascade resolution |

### Impact

- **openstrux-core**: validator + config packages
- **openstrux-spec**: Config and ADR-011 conformance fixtures
- Unblocks: lock-determinism, manifest-explain

## 4. Lock Determinism (`v0-6-0-lock-determinism`)

**Track:** Core compiler | **Repo:** openstrux-core + openstrux-spec

### Why

ADR-014 makes `snap.lock` a first-class input to compilation. Identical source + identical lock must yield byte-identical `mf.strux.json`. Without lock generation/consumption, manifest work and generator goldens sit on weaker ground ŌĆö there is no mechanism to freeze dependency state and verify deterministic builds.

The lock is part of the trust boundary. Manifest artifact hashes and certification scope are only meaningful when dependency versions are pinned.

### What changes

- `packages/lock/` ŌĆö `snap.lock` JSON schema, generation from validated + config-resolved AST, consumption during build, determinism verification
- Lock entries pin: adapter versions, config fingerprints, type registration hashes, hub artifact hashes (empty for v0.6.0)
- `sourceHash` in lock: SHA-256 of canonicalised source ŌĆö identity of the input
- Determinism tests: same source + same lock = byte-identical `artifactHash` in manifest output

### Capabilities

| Capability | Type | Description |
|---|---|---|
| `lock-generation` | New | Generates `snap.lock` from validated state, includes `sourceHash` |
| `lock-consumption` | New | Reads lock, constrains resolution |
| `lock-determinism` | New | Verifies source + lock = identical `artifactHash` |

### Impact

- **openstrux-core**: Lock package implemented
- **openstrux-spec**: Lock schema formalized, determinism test vectors
- Unblocks: manifest-explain

## 5. Manifest + Explain (`v0-6-0-manifest-explain`)

**Track:** Core compiler | **Repo:** openstrux-core + openstrux-spec

### Why

Validation and lock-determinism produce a clean, frozen model. The manifest is the compiled, fully resolved artifact ŌĆö `artifactHash` (identity of compiled output), certification scope, structured audit metadata (ADR-013). `lockRef` links the manifest back to the `sourceHash` in `snap.lock`. `--explain` is the near-term answer for human traceability, replacing deferred reverse translation.

### What changes

- `packages/manifest/` ŌĆö `mf.strux.json` with version, `artifactHash` (SHA-256 of compiled manifest content), certification scope, `lockRef` (referencing `sourceHash` from `snap.lock`), structured audit field (ADR-013)
- `--explain` CLI flag ŌĆö human-readable panel explanation from IR with source location tracing
- Golden fixtures for expected manifest output

### Capabilities

| Capability | Type | Description |
|---|---|---|
| `manifest` | New | `mf.strux.json` with `artifactHash`, `lockRef`, scope, audit (ADR-013) |
| `explain` | New | `--explain` CLI output from IR |
| `conformance-fixtures-golden` | New | Golden manifest fixtures |

### Impact

- **openstrux-core**: Manifest package + `--explain` CLI flag
- **openstrux-spec**: Golden fixtures
- Unblocks: target-ts

## 6. TypeScript Target Generator (`v0-6-0-target-ts`)

**Track:** Target generation | **Repo:** openstrux-core + openstrux-spec

### Why

The toolchain validates but produces nothing runnable. Target generation is the core value proposition ŌĆö AST -> real TypeScript. RFC-0001 establishes the normative adapter contract. A generator specification stabilizes the mapping rules before implementation.

### What changes

- `packages/generator/` ŌĆö adapter-based code generation engine
- TypeScript target adapter: `@type` records/enums/unions -> Prisma schema; `receive`/`respond` -> Next.js routes; `validate` -> Zod; `guard` -> access middleware
- `openstrux-spec/specs/modules/target-ts/generator.md` ŌĆö normative generator mapping spec
- RFC-0001: target adapter contract + Annex A (canonical source form for content hashing)
- Golden fixtures for generated TypeScript output

### Capabilities

| Capability | Type | Description |
|---|---|---|
| `generator` | New | Adapter-based code generation engine |
| `target-adapter-ts` | New | TypeScript/Prisma/Next.js/Zod adapter |
| `generator-spec` | New | Normative mapping spec in openstrux-spec |
| `conformance-fixtures-golden-ts` | New | Golden `.ts` and `.prisma` output |

### Impact

- **openstrux-core**: Generator package
- **openstrux-spec**: Golden fixtures + RFC-0001 + generator spec
- Enables: grant-workflow demo

## 7. Remaining Rods ŌĆö TypeScript (`v0-6-0-rods-target-ts`)

**Track:** Target generation | **Repo:** openstrux-core + openstrux-spec

### Why

Target-ts covers 6 rods for P0-P2. The remaining 12 are unhandled ŌĆö generator crashes on them. All 18 must be handled for completeness.

### What changes

- 12 additional TypeScript rod emitters:
  - **Tier 1** (real emitters, demo-capable): `transform`, `filter`, `write-data`, `call`, `split`, `pseudonymize`, `encrypt`
  - **Tier 2** (handled stubs, non-demo-capable): `group`, `aggregate`, `merge`, `join`, `window`
- Generator MUST NOT throw for any of the 18 rod types

### Exit criteria

| | Tier 1 | Tier 2 |
|---|---|---|
| Output | Real compilable TypeScript | `// STRUX-STUB` comment |
| Golden fixtures | Required | Not required |
| Demo-capable | Yes | No |
| Benchmark claims | Eligible | **Excluded** |
| Generator summary | Counted as implemented | Counted as stubbed |

### Capabilities

| Capability | Type | Description |
|---|---|---|
| `rods-target-ts` | New | All 18 rod emitters |
| `target-adapter-ts` | Modified | Dispatch table covers all 18 |

### Impact

- **openstrux-core**: `generator/adapters/typescript/rods/`
- **openstrux-spec**: Tier 1 golden fixtures only

## 8. Benchmarks (`v0-6-0-benchmarks`)

**Track:** Evidence | **Repo:** openstrux + openstrux-spec

### Why

v0.4-draft scored WARN on Principle 1 because benchmarks didn't exist. Without B001-B010, token compression claims are unverified and the release gate cannot pass.

> **Alpha-only evidence.** The manifesto requires 20+ benchmark cases across 3+ LLM families for a full Principle 1 gate. v0.6.0 delivers 10 cases against 1 LLM family ŌĆö sufficient for alpha-grade WARN-data but **not** for a full release gate. The remaining cases (B011-B030) and multi-model runs are deferred to v0.7.0.

### What changes

- **B001-B010**: 10 simple use-cases comparing `.strux` with equivalent TypeScript ŌĆö each defines a small problem and measures token compression ratio. Cases: typed record, enum, union, simple panel, RBAC endpoint, filter expression, config inheritance, named source reference, multi-rod pipeline, compliance rods.
- **Grant-workflow generation comparison**: Same prompt, two targets (direct TS vs `.strux`). Measures: input tokens, output tokens, wall-clock time, repair iterations.
- Hand-written TypeScript baselines
- v0.6.0 manifesto scorecard (PASS/WARN/FAIL per principle; WARN-data acceptable for alpha; Principle 1 MUST be WARN-data: "alpha-only evidence ŌĆö 10/20+ cases, 1/3+ LLM families")

### Capabilities

| Capability | Type | Description |
|---|---|---|
| `benchmark-cases-b001-b010` | New | 10 simple use-case token comparisons |
| `benchmark-generation-comparison` | New | Grant-workflow: direct-TS vs `.strux`, tokens + time |
| `manifesto-scorecard-v060` | New | Formal 7-principle review (alpha-only evidence) |

### Impact

- **openstrux**: benchmarks/cases + baselines + results
- **openstrux-spec**: Review scorecard
- No code changes to openstrux-core
