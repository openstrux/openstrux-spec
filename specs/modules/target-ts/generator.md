# OpenStrux — TypeScript Target Generator Specification

**Status:** Normative
**Spec version:** 0.6.0
**Implements:** RFC-0001 TypeScript Target Adapter

---

## 1. Overview

This document defines the normative mapping rules from validated OpenStrux AST nodes to TypeScript
artifacts. An adapter implementing this specification MUST produce output files that conform exactly
to these rules. Golden fixtures in `conformance/golden/target-ts/` are authoritative; this document
explains the rules from which they are derived.

Input to the generator: a validated AST (produced by `@openstrux/validator`) and a manifest
(`mf.strux.json`). Output: an array of `GeneratedFile` objects as defined in RFC-0001.

---

## 2. Output File Structure

```
types/
  <TypeName>.ts           One file per @type record, enum, or union
lib/
  schemas/
    <TypeName>.schema.ts  One Zod schema per @type record referenced in a validate rod
middleware/
  <PanelName>.guard.ts    One access middleware file per @panel with a guard rod
app/
  api/
    <panel-name>/
      route.ts            One Next.js App Router route file per @panel with a receive rod
prisma/
  schema.prisma           Single Prisma schema accumulating all models and enums
```

All paths are relative to the generator output root. The output root is determined by the caller
(not the generator).

---

## 3. Type Node Mappings

### 3.1 `@type` Record → TypeScript Interface + Prisma Model

A `TypeRecord` node with `n` fields produces:

**File:** `types/<Name>.ts`

```typescript
// generated by openstrux-generator — do not edit

export interface <Name> {
  <field1>: <MappedType1>;
  <field2>: <MappedType2>;
  // ...
}
```

**Appended to:** `prisma/schema.prisma`

```prisma
model <Name> {
  <field1> <PrismaType1>
  <field2> <PrismaType2>
  // ...
}
```

**Field type mappings:**

| OpenStrux `TypeExpr`                    | TypeScript type            | Prisma type       |
|-----------------------------------------|----------------------------|-------------------|
| `PrimitiveType { name: "string" }`      | `string`                   | `String`          |
| `PrimitiveType { name: "number" }`      | `number`                   | `Float`           |
| `PrimitiveType { name: "bool" }`        | `boolean`                  | `Boolean`         |
| `PrimitiveType { name: "date" }`        | `Date`                     | `DateTime`        |
| `PrimitiveType { name: "bytes" }`       | `Buffer`                   | `Bytes`           |
| `ContainerType { container: "Optional" }` | `<inner> \| null`       | `<inner>?`        |
| `ContainerType { container: "Batch" }`  | `<inner>[]`                | (not modelled)    |
| `ContainerType { container: "Map" }`    | `Record<string, <inner>>`  | `Json`            |
| `TypeRef { name: "<Name>" }`            | `<Name>`                   | `<Name>`          |
| `ConstrainedStringType { values }`      | `"v1" \| "v2" \| …`        | `String`          |
| `ConstrainedNumberType { min, max }`    | `number`                   | `Float`           |

`TypeRef` fields referencing an `TypeEnum` use the enum name for both TypeScript and Prisma types.

**Example — input:**

```
@type Proposal {
  id:           string
  title:        string
  submitter:    string
  description:  Optional<string>
  submitted_at: date
  status:       ReviewStatus
}
```

**Example — `types/Proposal.ts` output:**

```typescript
// generated by openstrux-generator — do not edit

export interface Proposal {
  id: string;
  title: string;
  submitter: string;
  description: string | null;
  submitted_at: Date;
  status: ReviewStatus;
}
```

**Example — Prisma model block (in `schema.prisma`):**

```prisma
model Proposal {
  id           String
  title        String
  submitter    String
  description  String?
  submitted_at DateTime
  status       ReviewStatus
}
```

### 3.2 `@type` Enum → TypeScript Enum + Prisma Enum

A `TypeEnum` node produces:

**File:** `types/<Name>.ts`

```typescript
// generated by openstrux-generator — do not edit

export enum <Name> {
  <variant1> = "<variant1>",
  <variant2> = "<variant2>",
  // ...
}
```

**Appended to:** `prisma/schema.prisma`

```prisma
enum <Name> {
  <variant1>
  <variant2>
  // ...
}
```

**Example — input:**

```
@type ReviewStatus = enum { draft, submitted, under_review, approved, rejected }
```

**Example — `types/ReviewStatus.ts` output:**

```typescript
// generated by openstrux-generator — do not edit

export enum ReviewStatus {
  draft = "draft",
  submitted = "submitted",
  under_review = "under_review",
  approved = "approved",
  rejected = "rejected",
}
```

**Example — Prisma enum block:**

```prisma
enum ReviewStatus {
  draft
  submitted
  under_review
  approved
  rejected
}
```

### 3.3 `@type` Union → TypeScript Discriminated Union

A `TypeUnion` node produces **only** a TypeScript discriminated union type. Unions are NOT Prisma
models (they represent abstract sum types; concrete leaf records are modelled separately).

**File:** `types/<Name>.ts`

```typescript
// generated by openstrux-generator — do not edit

export type <Name> =
  | { kind: "<tag1>"; value: <VariantType1> }
  | { kind: "<tag2>"; value: <VariantType2> }
  // ...
  ;
```

Each variant's `kind` discriminant is the tag name from the union definition. The `value` field
holds the resolved type expression for that variant.

**Example — input:**

```
@type SqlSource = union {
  postgres: PostgresConfig
}

@type DataSource = union {
  db: SqlSource
}
```

**Example — `types/SqlSource.ts` output:**

```typescript
// generated by openstrux-generator — do not edit

export type SqlSource =
  | { kind: "postgres"; value: PostgresConfig }
  ;
```

**Example — `types/DataSource.ts` output:**

```typescript
// generated by openstrux-generator — do not edit

export type DataSource =
  | { kind: "db"; value: SqlSource }
  ;
```

## 4. Rod Mappings

Rod mappings apply within a `@panel`. Each panel with relevant rods produces one route file in
`app/api/<panel-name>/route.ts`. Multiple rod emitters may write to the same route file; the
accumulation order follows rod declaration order within the panel.

### 4.1 `receive` Rod → Next.js App Router Route Stub

A rod with `rodType: "receive"` emits the HTTP handler skeleton.

**File:** `app/api/<panel-name>/route.ts` (created or augmented)

```typescript
// generated by openstrux-generator — do not edit

import { NextRequest, NextResponse } from "next/server";

export async function <METHOD>(req: NextRequest): Promise<NextResponse> {
  // TODO: implement <panel>/<rod-name>
  return NextResponse.json({ ok: true });
}
```

The HTTP method is derived from `rod.cfg["trigger"]`. If the trigger config specifies
`method: "POST"` (case-insensitive), the exported function name is `POST`. Supported methods:
`GET`, `POST`, `PUT`, `PATCH`, `DELETE`. Default: `GET`.

**Example — panel rod:**

```
intake = receive {
  trigger: http { method: "POST", path: "/proposals" }
}
```

**Example — `app/api/intake-proposals/route.ts`:**

```typescript
// generated by openstrux-generator — do not edit

import { NextRequest, NextResponse } from "next/server";

export async function POST(req: NextRequest): Promise<NextResponse> {
  // TODO: implement intake-proposals/intake
  return NextResponse.json({ ok: true });
}
```

---

### 4.2 `respond` Rod → Response Type Added to Route Stub

A rod with `rodType: "respond"` adds a typed response comment to the route file.

**Appended to:** `app/api/<panel-name>/route.ts`

```typescript
// Response type: <TypeName> (from respond rod: <rod-name>)
```

---

### 4.3 `validate` Rod → Zod Schema

A rod with `rodType: "validate"` produces a Zod schema derived from the referenced `@type` record.

**File:** `lib/schemas/<TypeName>.schema.ts`

```typescript
// generated by openstrux-generator — do not edit

import { z } from "zod";

export const <TypeName>Schema = z.object({
  <field1>: <ZodType1>,
  <field2>: <ZodType2>,
  // ...
});

export type <TypeName>Input = z.infer<typeof <TypeName>Schema>;
```

**Field type mappings to Zod:**

| OpenStrux `TypeExpr`                      | Zod expression              |
|-------------------------------------------|-----------------------------|
| `PrimitiveType { name: "string" }`        | `z.string()`                |
| `PrimitiveType { name: "number" }`        | `z.number()`                |
| `PrimitiveType { name: "bool" }`          | `z.boolean()`               |
| `PrimitiveType { name: "date" }`          | `z.coerce.date()`           |
| `PrimitiveType { name: "bytes" }`         | `z.instanceof(Buffer)`      |
| `ContainerType { container: "Optional" }` | `<inner>.nullable()`        |
| `ContainerType { container: "Batch" }`    | `<inner>.array()`           |
| `TypeRef { name }`                        | `z.lazy(() => <name>Schema)`|
| `ConstrainedStringType { values }`        | `z.enum([...])`             |
| `ConstrainedNumberType { min, max }`      | `z.number().min(min).max(max)` |

**Example — `lib/schemas/Proposal.schema.ts`:**

```typescript
// generated by openstrux-generator — do not edit

import { z } from "zod";

export const ProposalSchema = z.object({
  id: z.string(),
  title: z.string(),
  submitter: z.string(),
  description: z.string().nullable(),
  submitted_at: z.coerce.date(),
  status: z.nativeEnum(ReviewStatus),
});

export type ProposalInput = z.infer<typeof ProposalSchema>;
```

Note: `TypeRef` fields that resolve to a `TypeEnum` use `z.nativeEnum(<Name>)`.

### 4.4 `guard` Rod → Per-Route Access Middleware Stub

A rod with `rodType: "guard"` produces a per-route access guard wrapper.

**File:** `middleware/<PanelName>.guard.ts`

```typescript
// generated by openstrux-generator — do not edit

import { NextRequest, NextResponse } from "next/server";
import type { AccessContext } from "@openstrux/runtime";

export interface <PanelName>AccessContext extends AccessContext {
  purpose: "<purpose>";
  operation: "<operation>";
}

export async function withGuard(
  req: NextRequest,
  handler: (req: NextRequest, ctx: <PanelName>AccessContext) => Promise<NextResponse>
): Promise<NextResponse> {
  // TODO: implement access guard for <panel-name>/<rod-name>
  const ctx = {} as <PanelName>AccessContext;
  return handler(req, ctx);
}
```

The `purpose` and `operation` values are derived from the panel's `@access` block (`intent.purpose`
and `intent.operation`).

---

### 4.5 `store` Rod → Prisma Client Call Stub

A rod with `rodType: "store"` emits a Prisma client call stub appended to the route file.

**Appended to:** `app/api/<panel-name>/route.ts`

```typescript
// store: <rod-name>
// TODO: implement store — prisma.<model>.create({ data: ... })
```

---

### 4.6 `write-data` / `read-data` Rod → Prisma Query Stub

A rod with `rodType: "write-data"` or `"read-data"` emits a Prisma query stub appended to the
route file.

**Appended to:** `app/api/<panel-name>/route.ts`

```typescript
// write-data: <rod-name>
// TODO: implement write — prisma.<model>.upsert({ ... })

// read-data: <rod-name>
// TODO: implement read — prisma.<model>.findMany({ where: ... })
```

## 5. Raw-Expr Fallback

Any AST node that contains an expression the adapter cannot fully lower (e.g., complex expression
shorthand, unrecognised value expressions) MUST emit a TODO comment rather than failing.

**Rule:** For each unresolvable expression `<text>`:

```typescript
// TODO: implement expression: <text>
```

This keeps all generated output syntactically valid TypeScript while clearly flagging gaps.

---

## 6. Prisma Schema File

All Prisma blocks (models and enums) are accumulated across all `@type` definitions and emitted as
a single file `prisma/schema.prisma`.

**Header (always emitted first):**

```prisma
// generated by openstrux-generator — do not edit

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

**Body:** All `model` and `enum` blocks follow, in AST declaration order.

---

## 7. File Header Convention

Every generated file MUST begin with the comment:

```
// generated by openstrux-generator — do not edit
```

This is a single-line comment on line 1, before any imports or declarations.

---

## 8. Conformance

Conformance is verified by diffing actual generator output against golden files in
`conformance/golden/target-ts/`. The diff MUST be performed after:

1. Normalising whitespace (collapse runs of blank lines to a single blank line; trim trailing
   whitespace per line)
2. Sorting import statements alphabetically within each import group

A generator implementation passes conformance if the normalised diff is empty for all golden
fixtures.
