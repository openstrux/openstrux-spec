# OpenStrux — Next.js Rod Step Emitters

**Status:** Normative
**Spec version:** 0.6.0

---

## 1. Overview

Rod emitters in the Next.js adapter produce `ChainStep` objects rather than standalone string
snippets. The chain composer walks the panel's ordered rod sequence, calls each step emitter in
turn, and assembles the complete handler function body by combining all steps' imports and
statements.

---

## 2. ChainStep Type

```typescript
interface ChainStep {
  /** Imports this step needs at the top of the handler file. */
  imports: ImportDecl[]
  /** The code statement(s) for this step (one or more lines). */
  statement: string
  /** Variable name holding this step's output. */
  outputVar: string
  /** TypeScript type of the output variable. */
  outputType: string
}

interface ImportDecl {
  /** Named imports (e.g. ["NextRequest", "NextResponse"]). */
  names: string[]
  /** Module path (e.g. "next/server"). */
  from: string
  /** Whether to use `import type`. */
  typeOnly?: boolean
}

interface ChainContext {
  /** The containing panel. */
  panel: Panel
  /** Steps produced by earlier rods in the chain. */
  previousSteps: ChainStep[]
  /** Variable name holding the current input data. */
  inputVar: string
  /** TypeScript type of the current input data. */
  inputType: string
}
```

---

## 3. Chain Composer

The chain composer is responsible for:

1. Walking the panel's rod list in declaration order
2. Calling the appropriate `RodStepEmitter` for each rod
3. Deduplicating imports across all steps (same `from` + same `names` → merged)
4. Emitting a single `async function` with all statements in order

**Output shape for `handlers/<panel-name>.ts`:**

```typescript
// generated by openstrux-generator — do not edit

import { NextRequest, NextResponse } from "next/server";
// ... deduplicated imports from all steps

export async function <METHOD>(req: NextRequest): Promise<NextResponse> {
  <step1.statement>
  <step2.statement>
  <step3.statement>
  // ...
}
```

#### Scenario: Linear pipeline produces complete handler

- **WHEN** a panel contains `receive (POST) → validate → write-data → respond`
- **THEN** `handlers/<panel-name>.ts` SHALL export a complete async function with no TODO stubs

#### Scenario: Handler imports are deduplicated

- **WHEN** two steps both need `import { prisma } from "../lib/prisma.js"`
- **THEN** the output SHALL contain exactly one such import

---

## 4. Step Emitter Requirements

### Requirement: All rod types are handled

The adapter SHALL register a `RodStepEmitter` for every rod type in the 18-rod taxonomy.
Encountering an unregistered rod type SHALL produce a `STRUX-STUB` chain step, not a crash.

#### Scenario: Unknown rod type does not crash

- **WHEN** the chain composer encounters an unregistered rod type
- **THEN** it SHALL produce a `ChainStep` whose `statement` contains `STRUX-STUB`

## 5. Tier 1 Rod Step Emitters

### 5.1 `receive`

Emits the request body parsing statement. Always the first step.

```typescript
// imports: [{ names: ["NextRequest", "NextResponse"], from: "next/server" }]
// statement:
const body = await req.json();
// outputVar: "body", outputType: "unknown"
```

### 5.2 `validate`

Emits the Zod schema parse call.

```typescript
// imports: [{ names: ["<TypeName>Schema"], from: "../schemas/<TypeName>.schema.js" }]
// statement (inputVar from previous step):
const input = <TypeName>Schema.parse(<inputVar>);
// outputVar: "input", outputType: "<TypeName>Input"
```

#### Scenario: Validate step uses input from previous step

- **WHEN** a `validate` rod follows a `receive` rod (outputVar = "body")
- **THEN** the validate statement SHALL be `const input = ProposalSchema.parse(body);`

#### Scenario: No TODO stubs for validate with known type

- **WHEN** a `validate` rod references a `@type` record that exists in the AST
- **THEN** the chain step SHALL NOT contain any `// TODO` comment

### 5.3 `guard`

Emits an access context check. Also produces a separate `guards/<panel-name>.guard.ts` file (see
`generator.md §5.2`).

```typescript
// imports: [{ names: ["withGuard"], from: "../guards/<panel-name>.guard.js" }]
// statement:
const ctx = await withGuard(req, async (req, ctx) => {
// outputVar: "ctx", outputType: "<PanelName>AccessContext"
```

### 5.4 `write-data`

Emits a Prisma create or update call. The model name is lowercased from the input type name or
`cfg.target`.

```typescript
// imports: [{ names: ["prisma"], from: "../lib/prisma.js" }]
// statement (inputVar from previous step):
const result = await prisma.<model>.create({ data: <inputVar> });
// outputVar: "result", outputType: "<ModelName>"
```

#### Scenario: write-data uses previous step's output

- **WHEN** a `write-data` rod follows a `validate` rod (outputVar = "input", outputType = "ProposalInput")
- **THEN** the statement SHALL be `const result = await prisma.proposal.create({ data: input });`

### 5.5 `read-data`

Emits a Prisma findMany or findUnique call.

```typescript
// imports: [{ names: ["prisma"], from: "../lib/prisma.js" }]
// statement:
const result = await prisma.<model>.findMany();
// outputVar: "result", outputType: "<ModelName>[]"
```

### 5.6 `respond`

Emits the NextResponse return statement. Typically the final step.

```typescript
// imports: [] (NextResponse already imported by receive step)
// statement (inputVar from previous step):
return NextResponse.json(<inputVar>, { status: 201 });
// outputVar: "(returned)", outputType: "NextResponse"
```

### 5.7 `transform`

Emits a typed mapping function call.

```typescript
// statement:
const <outputVar> = transform<Name>(<inputVar>);
// outputVar: "<outputVar>", outputType: "<OutType>"
```

Where `transform<Name>` is a function the consumer is expected to provide (or the adapter
generates a typed stub function in the same file).

#### Scenario: Transform step has typed input and output

- **WHEN** a `transform` rod declares `in: Proposal` and `out: EligibilityRecord`
- **THEN** the chain step statement SHALL reference both types and the input variable

### 5.8 `filter`

Emits an array filter expression.

```typescript
// statement:
const <outputVar> = <inputVar>.filter((item) => /* predicate */);
// outputVar: "<outputVar>", outputType: "<ElementType>[]"
```

### 5.9 `split`

Emits a switch statement dispatching to named sub-chains.

```typescript
// statement:
switch (<inputVar>.kind) {
  case "eligible":
    // ...
    break;
  case "ineligible":
    // ...
    break;
}
```

#### Scenario: Split step has one case per named route

- **WHEN** a `split` rod defines routes `{ eligible, ineligible }`
- **THEN** the switch block SHALL contain `case "eligible":` and `case "ineligible":`

### 5.10 `call`

Emits a fetch call to an external endpoint.

```typescript
// statement:
const <outputVar> = await fetch("<endpoint>", { method: "<method>" }).then(r => r.json());
// outputVar: "<outputVar>", outputType: "unknown"
```

### 5.11 `encrypt`

Emits a compliance encryption wrapper call with JSDoc citing the accessed fields.

```typescript
/**
 * @compliance encrypt
 * @access scope.fieldMask: <field1>, <field2>
 */
const <outputVar> = await encrypt(<inputVar>, ctx);
```

#### Scenario: Encrypt step references scope fields in JSDoc

- **WHEN** the panel's `@access` block specifies `scope.fieldMask: ["ssn", "dob"]`
- **THEN** the JSDoc SHALL contain `@access scope.fieldMask: ssn, dob`

### 5.12 `pseudonymize`

Emits a compliance pseudonymization wrapper call with JSDoc citing the accessed fields.

```typescript
/**
 * @compliance pseudonymize
 * @access scope.fieldMask: <field1>, <field2>
 */
const <outputVar> = await pseudonymize(<inputVar>, ctx);
```

## 6. Tier 2 Rod Step Emitters (Stubs)

The following rod types produce `STRUX-STUB` chain steps. They do not crash the build.

| Rod type    | Stub statement                                                         |
|-------------|------------------------------------------------------------------------|
| `group`     | `// STRUX-STUB: group — <rod-name> — not implemented in v0.6`          |
| `aggregate` | `// STRUX-STUB: aggregate — <rod-name> — not implemented in v0.6`      |
| `merge`     | `// STRUX-STUB: merge — <rod-name> — not implemented in v0.6`          |
| `join`      | `// STRUX-STUB: join — <rod-name> — not implemented in v0.6`           |
| `window`    | `// STRUX-STUB: window — <rod-name> — not implemented in v0.6`         |

#### Scenario: Tier 2 stub is grep-able

- **WHEN** a Tier 2 rod appears in a panel
- **THEN** the generated handler SHALL contain the literal string `STRUX-STUB`
