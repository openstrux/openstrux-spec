# OpenStrux — Next.js Target Adapter Specification

**Status:** Normative
**Spec version:** 0.6.0
**Adapter name:** `nextjs`
**Stack:** TypeScript / Next.js App Router / Prisma / Zod

---

## 1. Overview

This document defines the normative mapping rules from validated OpenStrux AST nodes to
TypeScript artifacts for the Next.js target adapter. An implementation of this adapter MUST
produce output files that conform exactly to these rules. Golden fixtures in
`conformance/golden/target-nextjs/` are authoritative; this document explains the rules from
which they are derived.

Input to the adapter: a validated AST (produced by `@openstrux/validator`), a manifest
(`mf.strux.json`), and `ResolvedOptions`. Output from `emit()`: an array of `GeneratedFile`
objects with package-relative paths.

---

## 2. Output Package Structure

The package output directory defaults to `.openstrux/build/`. All paths below are relative to
that directory.

```
.openstrux/build/
  package.json              Adapter metadata, sub-path exports map
  tsconfig.json             strict, declaration, composite
  index.ts                  Root barrel — re-exports all types, schemas, handlers
  types/
    <TypeName>.ts           One file per @type record, enum, or union
  schemas/
    <TypeName>.schema.ts    One Zod schema per @type record referenced in a validate rod
    index.ts                Barrel — re-exports all schema objects
  handlers/
    <panel-name>.ts         One handler file per @panel with a receive rod
    index.ts                Barrel — re-exports all handler functions
  guards/
    <panel-name>.guard.ts   One guard file per @panel with a guard rod
  prisma/
    schema.prisma           Single Prisma schema — all models and enums
```

All paths in `GeneratedFile.path` are relative to the package output directory.

---

## 3. Stable Import Surface

`strux init` configures `tsconfig.json` path aliases so consumers import from a stable name
regardless of the physical location:

```json
{
  "compilerOptions": {
    "paths": {
      "@openstrux/build":   [".openstrux/build"],
      "@openstrux/build/*": [".openstrux/build/*"]
    }
  }
}
```

Consumer imports:

```typescript
import { Proposal }        from "@openstrux/build"
import { ProposalSchema }  from "@openstrux/build/schemas"
import { intakeProposals } from "@openstrux/build/handlers"
```

## 4. Type Node Mappings

### 4.1 `@type` Record → TypeScript Interface + Prisma Model

A `TypeRecord` node with `n` fields produces:

**File:** `types/<Name>.ts`

```typescript
// generated by openstrux-generator — do not edit

export interface <Name> {
  <field1>: <MappedType1>;
  <field2>: <MappedType2>;
  // ...
}
```

**Appended to:** `prisma/schema.prisma`

```prisma
model <Name> {
  <field1> <PrismaType1>
  <field2> <PrismaType2>
  // ...
}
```

**Field type mappings:**

| OpenStrux `TypeExpr`                    | TypeScript type            | Prisma type       |
|-----------------------------------------|----------------------------|-------------------|
| `PrimitiveType { name: "string" }`      | `string`                   | `String`          |
| `PrimitiveType { name: "number" }`      | `number`                   | `Float`           |
| `PrimitiveType { name: "bool" }`        | `boolean`                  | `Boolean`         |
| `PrimitiveType { name: "date" }`        | `Date`                     | `DateTime`        |
| `PrimitiveType { name: "bytes" }`       | `Buffer`                   | `Bytes`           |
| `ContainerType { container: "Optional" }` | `<inner> \| null`        | `<inner>?`        |
| `ContainerType { container: "Batch" }`  | `<inner>[]`                | (not modelled)    |
| `ContainerType { container: "Map" }`    | `Record<string, <inner>>`  | `Json`            |
| `TypeRef { name: "<Name>" }`            | `<Name>`                   | `<Name>`          |
| `ConstrainedStringType { values }`      | `"v1" \| "v2" \| …`        | `String`          |
| `ConstrainedNumberType { min, max }`    | `number`                   | `Float`           |

`TypeRef` fields referencing a `TypeEnum` use the enum name for both TypeScript and Prisma types.

**Example — input:**

```
@type Proposal {
  id:           string
  title:        string
  submitter:    string
  description:  Optional<string>
  submitted_at: date
  status:       ReviewStatus
}
```

**Example — `types/Proposal.ts`:**

```typescript
// generated by openstrux-generator — do not edit

export interface Proposal {
  id: string;
  title: string;
  submitter: string;
  description: string | null;
  submitted_at: Date;
  status: ReviewStatus;
}
```

**Example — Prisma model block:**

```prisma
model Proposal {
  id           String
  title        String
  submitter    String
  description  String?
  submitted_at DateTime
  status       ReviewStatus
}
```

### 4.2 `@type` Enum → TypeScript Enum + Prisma Enum

A `TypeEnum` node produces:

**File:** `types/<Name>.ts`

```typescript
// generated by openstrux-generator — do not edit

export enum <Name> {
  <variant1> = "<variant1>",
  <variant2> = "<variant2>",
  // ...
}
```

**Appended to:** `prisma/schema.prisma`

```prisma
enum <Name> {
  <variant1>
  <variant2>
  // ...
}
```

---

### 4.3 `@type` Union → TypeScript Discriminated Union

A `TypeUnion` node produces **only** a TypeScript discriminated union. Unions are NOT Prisma
models.

**File:** `types/<Name>.ts`

```typescript
// generated by openstrux-generator — do not edit

export type <Name> =
  | { kind: "<tag1>"; value: <VariantType1> }
  | { kind: "<tag2>"; value: <VariantType2> }
  ;
```

## 5. Rod Mappings

Rod mappings apply within a `@panel`. Each panel with a `receive` rod produces one handler file
at `handlers/<panel-name>.ts`. The handler is a complete implementation composed by the chain
composer from the panel's ordered rod sequence. See `rods.md` for the chain composition rules.

### 5.1 `validate` Rod → Zod Schema

A rod with `rodType: "validate"` produces a Zod schema file:

**File:** `schemas/<TypeName>.schema.ts`

```typescript
// generated by openstrux-generator — do not edit

import { z } from "zod";

export const <TypeName>Schema = z.object({
  <field1>: <ZodType1>,
  <field2>: <ZodType2>,
  // ...
});

export type <TypeName>Input = z.infer<typeof <TypeName>Schema>;
```

**Field type mappings to Zod:**

| OpenStrux `TypeExpr`                      | Zod expression                     |
|-------------------------------------------|------------------------------------|
| `PrimitiveType { name: "string" }`        | `z.string()`                       |
| `PrimitiveType { name: "number" }`        | `z.number()`                       |
| `PrimitiveType { name: "bool" }`          | `z.boolean()`                      |
| `PrimitiveType { name: "date" }`          | `z.coerce.date()`                  |
| `PrimitiveType { name: "bytes" }`         | `z.instanceof(Buffer)`             |
| `ContainerType { container: "Optional" }` | `<inner>.nullable()`               |
| `ContainerType { container: "Batch" }`    | `<inner>.array()`                  |
| `TypeRef { name }` (enum)                 | `z.nativeEnum(<Name>)`             |
| `TypeRef { name }` (record)               | `z.lazy(() => <name>Schema)`       |
| `ConstrainedStringType { values }`        | `z.enum([...])`                    |
| `ConstrainedNumberType { min, max }`      | `z.number().min(min).max(max)`     |

**Example — `schemas/Proposal.schema.ts`:**

```typescript
// generated by openstrux-generator — do not edit

import { z } from "zod";
import { ReviewStatus } from "../types/ReviewStatus.js";

export const ProposalSchema = z.object({
  id: z.string(),
  title: z.string(),
  submitter: z.string(),
  description: z.string().nullable(),
  submitted_at: z.coerce.date(),
  status: z.nativeEnum(ReviewStatus),
});

export type ProposalInput = z.infer<typeof ProposalSchema>;
```

### 5.2 `guard` Rod → Per-Panel Access Guard

A rod with `rodType: "guard"` produces a guard file:

**File:** `guards/<PanelName>.guard.ts`

```typescript
// generated by openstrux-generator — do not edit

import { NextRequest, NextResponse } from "next/server";
import type { AccessContext } from "@openstrux/runtime";

export interface <PanelName>AccessContext extends AccessContext {
  purpose: "<purpose>";
  operation: "<operation>";
}

export async function withGuard(
  req: NextRequest,
  handler: (req: NextRequest, ctx: <PanelName>AccessContext) => Promise<NextResponse>
): Promise<NextResponse> {
  const ctx = {} as <PanelName>AccessContext;
  return handler(req, ctx);
}
```

`purpose` and `operation` are derived from the panel's `@access` block.

## 6. Prisma Schema File

All Prisma blocks are accumulated across all `@type` definitions and emitted as a single file
`prisma/schema.prisma`.

**Header (always emitted first):**

```prisma
// generated by openstrux-generator — do not edit

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

**Body:** All `model` and `enum` blocks follow in AST declaration order.

## 7. Package Output (`package()` method)

The adapter's `package()` method produces ecosystem metadata and barrel entrypoints.

### 7.1 `package.json`

```json
{
  "name": "@openstrux/build",
  "version": "0.0.0",
  "private": true,
  "type": "module",
  "exports": {
    ".":          { "types": "./index.ts" },
    "./schemas":  { "types": "./schemas/index.ts" },
    "./handlers": { "types": "./handlers/index.ts" }
  }
}
```

### 7.2 `tsconfig.json`

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "declaration": true,
    "composite": true,
    "outDir": "./dist"
  }
}
```

### 7.3 Root `index.ts`

Re-exports all generated types:

```typescript
// generated by openstrux-generator — do not edit

export type { Proposal } from "./types/Proposal.js";
export { ReviewStatus } from "./types/ReviewStatus.js";
// ... one export per type file
```

### 7.4 `schemas/index.ts`

Re-exports all generated schemas:

```typescript
// generated by openstrux-generator — do not edit

export { ProposalSchema } from "./Proposal.schema.js";
export type { ProposalInput } from "./Proposal.schema.js";
// ...
```

### 7.5 `handlers/index.ts`

Re-exports all generated handler functions:

```typescript
// generated by openstrux-generator — do not edit

export { POST as intakeProposals } from "./intake-proposals.js";
// ...
```

## 8. File Header Convention

Every generated file MUST begin with:

```
// generated by openstrux-generator — do not edit
```

---

## 9. Raw-Expr Fallback

Any AST node the adapter cannot fully lower MUST emit a `STRUX-STUB` comment rather than
failing:

```typescript
// STRUX-STUB: <rod-type> — <rod-name> — <reason>
```

This keeps all generated output syntactically valid TypeScript while clearly marking gaps.

---

## 10. Conformance

Conformance is verified by diffing actual adapter output against golden files in
`conformance/golden/target-nextjs/`. The diff MUST be performed after applying the normalisation
rules defined in `specs/generator/generator.md §8`.
